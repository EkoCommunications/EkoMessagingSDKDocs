Concepts

A good rule of thumb is:

  • LiveData classes detemines when to query;
  • Transactions determine how to query;
  • Coordinator makes sure the query goes through to the server properly.

LiveData

Responsible for returning data back to the developer. There are two types of LiveData: object and collection. Object is just a single object, collection is multiple objects. Both LiveData types are just a RLMResult wrapper.

As long as a LiveObject or LiveCollection is in memory, it is assumed that the query is active, and the client should do as much as possible to keep it in sync with the server, including querying for fresh data from the server, if necessary. LiveData should contain a Transaction instance to make queries.

There should be a maximum number of active LiveData objects, to prevent the case of too many network requests caused by active LiveData queries.

Repository

Repository should be the place to instantiate a LiveData object with corresponding database query and transaction, and return that LiveData to the user.

Transactions

Responsible for making the server query and managing server responses. It provides a consistent interface for querying for objects and collections to be managed by LiveData classes.

Request

Requests are responsible for gathering all paramaters from the developer and starting the transations.

Naming Conventions

  • Id and Ids (not ID and IDs);
  • (for blocks) completion (not complete);
